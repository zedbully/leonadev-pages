# 拿到 DEX 和 ELF，为什么仍还原不出核心算法？御盾 Android 加固四级恢复实测

原文已发布在御盾技术内容中心：[查看完整实测与证据链](https://dun.leonadev.com/article/yudun-android-dex-elf-algorithm-recovery-four-level-test)。

运行期工具能够写出 DEX 或可解析 ELF，只说明拿到了可继续分析的程序材料，不足以证明核心算法已经还原。真正的算法还原需要恢复实现脱离原应用后，仍能对相同输入、关键状态和异常条件产生等价结果。

## 四级恢复判定

| 层级 | 判定问题 | 本次观察 |
|---|---|---|
| L1 材料生成 | 工具是否输出了 DEX 或 native 镜像？ | 两类材料均有输出。 |
| L2 格式有效 | 标准解析器能否稳定读取？ | 部分 DEX 有效；后备 native 载体为有效 ARM64 ELF。 |
| L3 程序结构 | 是否恢复类、控制流、模块或导出面？ | DEX 恢复了更多类与部分控制流；ELF 直接业务语义有限。 |
| L4 算法等价 | 能否独立重现原始核心计算？ | 本次没有形成等价实现。 |

## 为什么文件成功不等于算法成功

深度内存搜索会同时增加有效材料和异常候选。如果只统计输出文件数量，重复映射、残片和伪阳性都会被写进“恢复成功率”。因此 DEX 需要去重、格式解析和恢复后再反编译；native 镜像需要检查目标架构、程序头、段与加载信息。只有通过这些检查，材料才能从 L1 进入 L2。

即使文件有效，也只是进入结构分析阶段。本次运行期 DEX 恢复增加了类、入口轮廓和部分控制流可见性，但核心计算仍跨越桥接与 native 分发。native 侧取得的有效载体已剥离符号，对外导出集中在少量桥接与注册类别。恢复材料有分析价值，却没有直接构成可独立运行的原始算法。

## 脱敏测评经过

1. 先在干净环境确认同一加固候选能够安装、连续冷启动并完成一条关键业务路径。
2. 对原包执行结构、Manifest、DEX、反编译、native 和资源等静态基线检查。
3. 分别执行标准与深度 DEX 恢复，对结果去重并由独立解析器复核。
4. 将有效 DEX 再次反编译，比较原包与恢复集合的类和控制流差异。
5. 观察运行期 native 映射，区分连续内存镜像与映射后备载体。
6. 使用独立 ELF 视角核对架构、加载信息、符号、安全属性和导出面。
7. 最后使用输入、状态、异常语义和独立复现四项门槛判断算法等价。

## App 加固 PoC 应该怎样写结论

推荐写法是：“在已执行的标准与深度恢复、恢复后反编译和 ELF 结构复核中，程序材料达到 L1-L3 的不同程度，未形成 L4 核心算法等价实现。”这句话同时保留了恢复事实、正向结果和测试边界。

不推荐使用“完全无法 dump”“拿到 SO 就算破解”“工具报错所以防护通过”等说法。前两种混淆材料与语义，后一种可能把工具版本、权限或传输问题冒充产品能力。
